I Backfilled the Forecast Before I Used It as a Feature

Rebuilding historical time-series forecasts so a hybrid model can actually learn from them

View Repository Explore Method

Overview

This project forecasts weekly sea turtle rescue counts across 29 Kenyan capture sites. The modeling system combined a per-site seasonal time-series baseline with a supervised model designed to learn cross-site corrections.

The key issue was not the ensemble itself. The real challenge was making the time-series forecast a valid feature for the supervised model.

A forecast cannot be used as a training feature if it only exists for the final prediction year. So the forecast was rebuilt historically using expanding-window forecasts for earlier years as well.

Architecture

The Core Idea

The supervised model needed to learn from rows where both of these existed:

  • a time-series forecast baseline
  • the actual observed rescue count

If the forecast is generated only for 2019, then it exists only in the test period. The correction model has nothing to train on.

A forecast becomes a usable feature only after its history is recreated.

How the feature was backfilled

Per-site STL + ETS forecasts were generated for each target year using only data available before that year:

  • forecast 2014 using pre-2014 data
  • forecast 2015 using pre-2015 data
  • forecast 2017 using pre-2017 data
  • forecast 2018 using pre-2018 data
  • forecast 2019 using pre-2019 data
md_19 <- forecast_year(df_ts, 2019)
md_18 <- forecast_year(df_ts, 2018)
md_17 <- forecast_year(df_ts, 2017)
md_15 <- forecast_year(df_ts, 2015)
md_14 <- forecast_year(df_ts, 2014)

ts_forecasts <- rbind(md_19, md_18, md_17, md_15, md_14)

How it entered the supervised model

df <- df %>%
    left_join(ts_forecasts, by = c("CaptureSite", "year", "week_"))

names(df)[which(names(df) == "Capture_Number")] <- "ts_forecast"
df$ts_forecast[is.na(df$ts_forecast)] <- 0

This allowed CatBoost and XGBoost to learn when site-level and seasonal features should override or trust the baseline forecast.

Why this matters

  • prevents using a meta-feature that only exists in the test period
  • creates historically honest training rows for the correction model
  • turns a time-series model from final output into a learnable input
  • generalizes to hybrid forecasting systems beyond this project

Selected implementation details

  • per-site STL + ETS seasonal forecasting
  • 2016 excluded because of structural data distortion
  • forecast outlier capping with guard limits
  • final blend: 70% TS baseline + 30% CatBoost correction

Key Takeaway

I backfilled the forecast before I used it as a feature.

A hybrid model can only learn from a forecast input if that input is generated historically under the same information constraints.

Built as a public project showcase for historically honest forecast-as-feature design.
